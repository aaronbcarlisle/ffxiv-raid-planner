### 4. Missing Focus Indicators on Interactive Elements

**Files affected:**
- `/home/serapis/projects/ffxiv-raid-planner/frontend/src/components/history/WeeklyLootGrid.tsx`
- `/home/serapis/projects/ffxiv-raid-planner/frontend/src/components/loot/WhoNeedsItMatrix.tsx`

**Current Implementation:**
Interactive cells in WeeklyLootGrid have `tabIndex={0}` and keyboard handlers, but no visible focus ring:

```tsx
// WeeklyLootGrid.tsx - Line 419
className={`min-w-[100px] flex-1 px-3 py-2 border-l border-border-subtle hover:bg-surface-elevated/50 transition-colors ${isClickable ? 'cursor-pointer' : ''}`}
```

**Recommended Fix:**
Add focus-visible styles:
```tsx

```

**Impact:** Keyboard users cannot see which element has focus (WCAG 2.4.7 violation).

### 5. Inconsistent Border Radius Patterns

**Analysis:** The codebase uses multiple border radius values inconsistently:

| Pattern | Usage | Files |
|---------|-------|-------|
| `rounded` | Most common, ~4px | Throughout |
| `rounded-lg` | Cards, modals | Modal, PlayerCard |
| `rounded-md` | Buttons, inputs | Button, Input |
| `rounded-full` | Avatars, indicators | UserMenu, badges |

**Inconsistencies Found:**
- `/home/serapis/projects/ffxiv-raid-planner/frontend/src/components/loot/LootLogPanel.tsx` uses `rounded` for container but `rounded-lg` would match other cards
- Badge components mix `rounded` and `rounded-md`

**Recommended Action:** Establish a border radius scale in the design system:
- `rounded-sm` (2px) - small badges
- `rounded` (4px) - buttons, inputs, small containers
- `rounded-lg` (8px) - cards, modals
- `rounded-xl` (12px) - large containers (rarely used)
- `rounded-full` - avatars, pills

## Accessibility Audit Summary

### WCAG 2.1 AA Compliance Status

| Criterion | Status | Notes |
|-----------|--------|-------|
| 1.4.3 Contrast (Minimum) | PASS | Text colors meet 4.5:1 ratio |
| 1.4.11 Non-text Contrast | PARTIAL | Some interactive states may not meet 3:1 |
| 2.1.1 Keyboard | PARTIAL | Most elements accessible, some missing focus indicators |
| 2.4.7 Focus Visible | PARTIAL | Inconsistent focus ring implementation |
| 4.1.2 Name, Role, Value | PARTIAL | Some icon buttons missing aria-labels |

### Recommended Accessibility Improvements

1. **Add focus-visible styles** to all interactive elements (WeeklyLootGrid cells, matrix buttons)
2. **Add aria-labels** to all icon-only buttons
3. **Add aria-live regions** for toast notifications (may already exist via library)
4. **Test with screen reader** to verify announcement order and content

## Responsive Design Audit

### Breakpoint Coverage

The codebase uses consistent Tailwind breakpoints:
- Mobile: Default (< 640px)
- Tablet: `sm:` (640px+), `md:` (768px+)
- Desktop: `lg:` (1024px+), `xl:` (1280px+)
- Ultra-wide: `2xl:` (1536px+)

### Issues Found

1. **WeeklyLootGrid** - May overflow on small screens due to fixed minimum cell widths
2. **WhoNeedsItMatrix** - Table layout requires horizontal scroll on mobile
3. **TeamSummaryEnhanced** - Grid layout collapses well but table headers crowd on mobile

### Recommendations

1. Consider card-based layout for WeeklyLootGrid on mobile
2. Add horizontal scroll indicator for matrix tables
3. Test touch targets (minimum 44x44px) on all interactive elements

## Component Pattern Analysis

### Well-Implemented Patterns

1. **Semantic Color Tokens**: Extensive use of `text-`, `bg-`, `border-` semantic classes
2. **Role Colors**: Consistent use of `role-tank`, `role-healer`, etc.
3. **Status Colors**: Proper use of `status-success`, `status-warning`, `status-error`
4. **Membership Colors**: Good abstraction for `membership-owner`, `membership-lead`, etc.
5. **Material Colors**: Semantic tokens for `material-twine`, `material-glaze`, `material-solvent`

### Patterns Needing Improvement

1. **Inline Styles**: Several components use `style={{ color: ... }}` instead of Tailwind
2. **Computed Colors**: Dynamic colors based on calculations use hex values
3. **Animation Classes**: Inconsistent use of `transition-*` utilities

## Conclusion

The FFXIV Raid Planner frontend demonstrates a mature design system with thoughtful color token organization and generally consistent component patterns. The main areas requiring attention are:

1. **Hardcoded colors** in dynamic styling scenarios
2. **Missing accessibility attributes** on some interactive elements
3. **Inconsistent use of primitives** in some newer components

The existing design system infrastructure is excellent - the primary work needed is ensuring all components fully leverage it. The codebase is well-positioned for these improvements due to the centralized token system in index.css.
